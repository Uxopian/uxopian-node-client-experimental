## 1. The three API surfaces (and which auth each needs) ✅

| Surface | Base | Auth | Use |
|---|---|---|---|
| **Core REST** | `https://<host>/core` (top-level `/core`) | **JWT token** header `token: <jwt>` | Full CRUD: documents, classes, tags, files, search. The one to use for automation. |
| **GUI REST** | `https://<host>/gui/rest/...` | session **cookie** (Puppeteer login) | Reads (config/classes, scripts, documents). **Writes need a `policy` CSRF token** the admin UI supplies — avoid; use Core. |
| **uxopian-ai gateway** | `https://<host>/gui/plugins/<scope>/gateway/uxopian-ai` | session **cookie** | Prompts/goals admin + chat (the AI side). |

- ⚠️ `https://<host>/gui/plugins/<scope>/gateway/core` is **locked** (401 / login page). Use the **top-level `/core`** instead.
- The Puppeteer GUI cookie works for `/gui/*` and the uxopian-ai gateway, **but NOT for `/core`** — `/core` needs the JWT.

## 2. Core API authentication ✅

```bash
# POST {base}/rest/authentication  ->  { "value": "<JWT>" }
curl -X POST "https://iris.demos.uxopian.com/core/rest/authentication" \
  -H "Content-Type: application/json" \
  -d '{"user":"system","password":"<pw>","scope":"IRIS"}'
# then send:  -H "token: <JWT>"
```
- JWT ~273 chars; `profiles: SYSTEM_ADMIN, ADMIN`; **expires ~1h** (re-auth in long jobs).
- Reference example: `/Users/aescaffre/flowertoken.py`. Field names: `user`, `password`, `scope`.

## 3. Documents (Core REST) ✅

- **Create**: `POST /core/rest/documents` — body is a **JSON ARRAY** of document objects. Returns the created docs (with ids).
  ```json
  [{"id":"FriendlyId","name":"Human name","category":"DOCUMENT",
    "data":{"classId":"Script","ACL":"acl-readonly"},
    "files":[{"id":"tmp_..."}],
    "tags":[{"name":"RegistrationOrder","value":["900"],"readOnly":false}]}]
  ```
  - You **can set a friendly `id`** (e.g. `ContractIntelligenceCockpit`) — it sticks. Omit it to get a UUID.
  - `files:[{id: <tmpFileId>}]` attaches uploaded temp-file content on create. ⚠️ The returned `files[].size` shows **`0`** even when content is present — it's a metadata display quirk, not empty (verified: 102 KB content served back fine).
- **Read**: `GET /core/rest/documents/{id}` (returns an array `[ { files, id, data:{classId,status,version,owner,ACL}, tags, category, name, currentVersion } ]`).
- **Content**: `GET /core/rest/documents/{id}/files/{fileId}/content` (the `?index=0` variant 405s — use the file id in the path).
- **Delete**: `DELETE /core/rest/documents/{id}`.
- **Search**: `POST /core/rest/documents/search`
  ```json
  {"selectClause":{"fields":["name","classid"]},
   "filterClauses":[{"@class":"com.flower.docs.domain.search.AndClause",
     "criteria":[{"name":"classid","type":"STRING","operator":"EQUALS_TO","values":["Script"]}]}],
   "max":50}
  ```
  Returns `{found, results:[{fields:[{name,value}], id}]}`.

## 4. Temporary files (Core REST) ✅

POST /core/rest/files/tmp   (multipart -F "file=@path")  -> {"id":"tmp_<uuid>", "Name", "FormatCode", "size", ...}
GET  /core/rest/files/tmp/{id}        # recover

## 5. Scripts = JS plugins = `Script`-class documents ✅

- A FlowerDocs "script" (JSAPI plugin) is a **document of class `Script`**. The GUI serves it client-side at **`GET /gui/rest/scripts/{id}`** (read-only; POST/PUT/DELETE there → 405).
- **It loads in the client when it carries a `RegistrationOrder` integer tag** (lower = loaded earlier; `CustomActions`=1). No `RegistrationOrder` ⇒ stored but not loaded.
- **Deploy recipe** (no server access; just the Core JWT): upload JS to `/rest/files/tmp` → `POST /rest/documents` with `classId:"Script"`, friendly `id`, `files:[{id:tmpId}]`, tag `RegistrationOrder`. ✅ (deployed `ContractIntelligenceCockpit` this way.)
- ✅ **VERIFIED END-TO-END — full API deploy works (no server, no admin UI):**
  1. Auth `/core` with **`scope:"IRIS"`** (the token's scope = the scope the doc lands in — using `FD` would put it on the wrong scope).
  2. `POST /core/rest/files/tmp` (multipart) → `tmp_<id>`.
  3. `POST /core/rest/documents` (array): `{id:"ContractIntelligenceCockpit", name, category:"DOCUMENT", data:{classId:"Script", ACL:"acl-readonly"}, files:[{id:tmpId}], tags:[{name:"RegistrationOrder", value:["900"]}]}`. Friendly `id` sticks; content attaches.
  4. **Purge the GUI script cache:** `DELETE /gui/rest/caches` (cookie session, clearAll). `GET /gui/rest/caches` lists the GUI caches incl. **`IRIS-Script`** and **`IRIS-GUIConfiguration`** — *these* hold the loaded-script list (NOT the `/core/rest/caches` data caches). Skip this and the GUI keeps its cached set and never loads the new doc.
  5. **Full page reload** (a hash change does NOT re-inject scripts). Verified: loaded-script count goes 49→50, ours present.
- 🔑 **In-script registration API = the GLOBALS `MenuShortcutsAPI` / `ContextualMenuAPI`, each `.get().registerForLoad(function(api){...})`** — NOT `JSAPI.get().registerForLoad(...)` (that throws `JSAPI.get(...).registerForLoad is not a function` and silently kills the plugin). The `api` arg IS the container: toolbar `api.addCircled(id,icon,color,name,desc,cb)` / `api.addIconized(...)`; contextual `api.add(groupId,id,icon,label,cb)` + `api.getSelected()` / `api.getCategory()`. Confirmed from the working `env-taskAvailableContextualMenu` script.
- ⚠️ `JSAPI.get()` is **empty on Home**; the shortcut/contextual containers populate only inside a component/search/document view — that's where the shortcut renders (and where to verify). **VERIFIED:** after the fix, the cockpit script loads and the **"Contract Intelligence" toolbar shortcut + contextual action register with no JS errors.**

## 5b. JSAPI entry points — where a plugin's trigger renders (verified) 🔑

`JSAPI.get()` exposes many `*API` globals (`MenuShortcutsAPI`, `ContextualMenuAPI`, `ComponentFormJSAPI`, `ActionFactoryAPI`, `DocumentAPI`, …). Triggers are **per-context** — pick the right one for where the user is:
- **Search/browse toolbar:** `MenuShortcutsAPI.get().registerForLoad(fn(api){ api.addCircled(...) })` → a circular button in the search view's menu bar. (Verified it renders on a search view.) ⚠️ Don't use it to act on "the open document" — on a search view there is none.
- **Search-result rows:** `ContextualMenuAPI.get().registerForLoad(fn(api){ api.add(group,id,icon,label,cb); api.getSelected() })` → an action on selected rows. **Most reliable way to act on a chosen document — but you need a search that returns rows of that class.**
- **Open-document form (✅ VERIFIED — this is what the NDA cockpit uses):** `JSAPI.get().registerForComponentChange(fn(componentFormAPI, component, phase){ if(component.getCategory()=='DOCUMENT'){ var a=JSAPI.get().getActionFactoryAPI().buildResponsive(id,label,icon,0,cb); componentFormAPI.getActions().getHeaderActions().add(a); } })` (docs p296). `registerForComponentChange` **is** a method of `JSAPI.get()`. **Renders as a named action ("Contract Intelligence") in the open document's action UI — user-confirmed present.** `component.getClassId()` gives the class to guard on. NOTE: headless Puppeteer struggled to locate/click it (it's in the doc action set / overflow), but it IS there for a real user — don't trust an automated "not found" here.
- **Search form actions:** `JSAPI.get().registerForSearchOpen(fn(searchFormAPI,id){ searchFormAPI.getFooterActions().add(...) })`; `getActionFactoryAPI().buildTextual/buildIcon/buildResponsive`.
- The NDA cockpit launches from the open-document action above; a contextual action on an NdaContract **search row** is a complementary entry point (needs a native search screen — part of the iterative native-UI loop).

## 5c. "Clauses as data" — query the relevant subset via tool-using prompts ✅ (verified)

Don't dump clause content into prompts, and don't rely on client-side JSAPI search (no working-script precedent here; the cockpit's `getSearchAPI().searchDocuments()` assumption returns empty). Instead: store clauses as `NdaClause` documents (tags incl. `ClauseText`, `SimilarityGroup`, `LifecycleStatus`, `RiskScore`, `SourceContractId`), and let the **LLM query only the relevant subset via the shipped FlowerDocs search TOOLS** (function-calling).
- Deploy the prompt with **`requiresFunctionCallingModel: true`** + provider/model that supports tools (`openai`/`gpt-4o`). The shipped `flowerdocs`-tagged tools (`getFlowerDocsDataModel`, `buildTextCriterion`/`buildNumericCriterion`, combiners, `searchDocuments`, read tags) are auto-merged into the request's tool catalogue — works on **both** `/requests/stream` and `/requests`.
- Prompt instructs: "use the tools to find `NdaClause` where `SourceContractId == [[${sourceContractId}]]` and `LifecycleStatus in (Obsolete, Missing or weak)`, read `ClauseText`/`ClauseFamily`, then analyze." The host only passes light context in the payload (`sourceContractId`, `similarityGroup`, `documentId`) — **not** clause data.
- ✅ Verified: returns the real curated clauses (C00112 pré-RGPD, C00120 Safe Harbor; risk 81/100) — traceable by `ClauseId`, bounded, no dumping. Trade-off: slower (multi-step tool calls) — show a loading/skeleton state.
- Goals still 400 on unresolved Thymeleaf vars — use a **direct PROMPT** input (`type:"PROMPT"`), not a GOAL.
- ⚠️ **EXACT `flowerdocs` tool names (from source `FlowerDocsSearchService.java`, not the older notes):** Step-0 is **`getTaskClassAndTagClassesDescriptions`** (NOT `getFlowerDocsDataModel`); criteria builders are **`buildCriterionString` / `buildCriterionNumber` / `buildCriterionClass` / `buildCriterionDate`** (NOT `buildTextCriterion`/`buildNumericCriterion`); combiners **`buildAndClause`/`buildOrClause`** take a `List<Criterion>` → return ONE `SearchFilterDto`; **`buildAndClauseFromClauses`/`buildOrClauseFromClauses`** nest a `List<SearchFilterDto>`; **`searchDocuments(List<SearchFilterDto>)`** takes a LIST. A bare `Criterion` MUST be wrapped in a clause before search. `searchDocuments` **catches every exception/empty and returns bare `null`** (ambiguous) — so a mis-shaped arg looks identical to "no results"; prompts must state the wrapping contract explicitly and retry once on null before concluding. Naming the wrong/nonexistent tools (the old `getFlowerDocsDataModel` etc.) is what made the NDA compare prompts give up with "erreur technique".
- ⚠️ **`CriterionBuilder` default `type` is `STRING`** (verified in bytecode), so `buildCriterionString`/`buildCriterionClass` emit `type=STRING` and search fine. But a hand-written `POST /core/rest/documents/search` criterion with **`type:null` → `500 T00104` "Components cannot be found due to a technical error"**; always set `"type":"STRING"` on REST criteria. (Same `T00104` family as INT `orderClauses` / nested aggregations — it is this instance's generic "search engine couldn't run that" error.) Live-verified: `ContractGroup EQUALS_TO "GRP-NN"` (type STRING) returns the FR/EN twin pair (e.g. GRP-10 → NDA_EN_10 + NDA_FR_10); the search SelectClause only returns `classid/<criterion tags>/acl`, but the tool then does `documentService.get(ids)` so the LLM receives FULL docs with all tags — no re-search needed.

## 6. Classes & tags (Core REST)

- `GET /core/rest/tagclass` ✅ lists **all** tag classes; `/rest/tagclass/{ids}` get; `DELETE /rest/tagclass/{ids}`.
- **Create tag class** = `POST /core/rest/tagclass` with an **array** body. Re-POSTing an existing id → **500 `F00903` "already exists"** (create is NOT an upsert). Valid `type` values: `STRING, TEXT, INT, CHOICELIST, DATE, BOOLEAN, ICON` (NOT `INTEGER`).
- **Update tag class** ✅ (verified, doc p909) = `POST /core/rest/tagclass/<IDS>` — **id(s) in the PATH**, **array** body (the full tag-class object: `data`/`type`/`displayNames`/`searchable`/`id`). Returns 200 + the updated object. ⚠️ Gotchas confirmed live: `PUT /rest/tagclass[/{id}]` → **405**; `POST /rest/tagclass` (no id, existing) → 500 exists; `POST /rest/tagclass/{id}` with a **single object** (not array) → **400**. So: array-in-body, id-in-path, POST verb. This is the way to fix a tag's `displayNames` (e.g. machine-id → human label) without delete/recreate.
- **Document class create/update** = `POST /core/rest/documentclass` (array). Needs `category:"DOCUMENT"` + `active:true` (else F00204 "no category"). Class update is **full-replace** (resend every `tagReference` + flag, or they clear) — same caution as documents (doc p877).
- Tag grouping into form "blocks" = **TagCategory** objects (own REST `/core/rest/tagcategory`) + the class's `tagCategories` list. ✅ verified live shape:
  - A TagCategory = `{"id":"NdaIdentite","displayNames":[{value,language}],"tags":["TagName1",...],"icon":"flower-button header-icon fas fa-file flat-purple","visible":true,"inline":false,"reduced":false}`. The **`tags` array is the membership** (which of the class's tags fall in this block) — the `tagReference` objects on the class do NOT carry a category field.
  - Create = `POST /core/rest/tagcategory` (array). Update = `POST /core/rest/tagcategory/{ids}` (array). (Same verb pattern as tagclass.)
  - The class's `tagCategories` is a **list of category IDs** (strings), e.g. `["NdaIdentite","NdaEvaluation"]`. Set it via the class full-replace (`POST /core/rest/documentclass/{id}`, array body — verified 200). A TagCategory not listed in the class's `tagCategories` does NOT render.
  - Per-tag `readonly`/`mandatory` are flags on the class `tagReference` objects (full-replace). Verified: setting `readonly:true` on the AI-computed tags renders them read-only in the form; opening a doc shows the grouped blocks with human field labels (from tag `displayNames`) + the ARender preview beside the form.
- `GET /core/rest/componentclass` → 404; the correct resources are `/rest/documentclass`, `/rest/tagclass`, `/rest/tagcategory` (per-component-type, not a unified "componentclass").

## 6b. GUIConfiguration / native UI as data (Core REST) ✅ — verified on IRIS

GUI config (search screens, forms, home dashboards, shortcuts, menus) lives in **`GUIConfiguration`-class documents** whose single file is a **Spring-bean XML**. They are loaded into one shared GWT bean context; a bean error in one doc can break the whole GUI for everyone — so on a shared instance keep new docs **self-contained + namespaced + additive**.

- **Fetch a config doc's XML:** `GET /core/rest/documents/{id}` returns an **array** → `[0].files[0].id` → `GET /core/rest/documents/{id}/files/{fileId}/content`. Known IRIS doc ids: `documentSearch`, `gui-solution` (main wiring: arenderConfig/appInfo/SearchableFieldCatalog/`ComponentPropertiesConfiguration`/`componentActivityConfigurations`), `form-configuration`, `sol-shortcuts`, `env-search-criteria`, `env-enveloppes-search`, `env-home` (~17 GUIConfiguration docs total).
- **Search screen** = a `com.flower.docs.gui.client.search.ComponentSearchPresenter` bean (`scope="prototype"`). Key properties (all verified live): `responsePresenterProvider`=`TableSearchResponsePresenterProvider` (sortable table; **no per-row JS card API in table mode** — use tag columns or switch to Card/Switcher provider for `${field}` templates); `categorySelectorPresenter`=`FakeCategorySelectorPresenter` pinned to `DOCUMENT`; `keywordCriteriaPresenter` (usually `enabled=false`); `advancedCriteriaPresenter`=`AdvancedCriteriaPresenter` whose `fixedCriterionPresenters` is a `<list>` of `<ref bean=...>` to criterion beans; `hiddenColumns` (hide `status`/`classid`); and `hiddenRequest`=`SearchRequest` carrying `selectClause.fields` (= **result columns, in order**), `orderClauses`→`OrderClause{name,ascending}` (doc p…14845 "default sort order"), `aggregation`→`FieldAggregation{field}` (one-click facet, verified live in env-home), and `filterClauses` (the **invisible scope**: `Criterion name=classid EQUALS_TO [classes]`).
- **Search gotchas (verified live on IRIS, 2026-06-05 — both surfaced as "The search could not be executed" / HTTP 500 in the GUI):**
  - **`orderClauses` on an INT tag fails** → `500 T00104` "Components cannot be found due to a technical error". INT tags (e.g. `ClauseOrder`, `RiskScore`, `NdaGlobalScore`) are NOT server-sortable here. Order by a **STRING** field instead (or omit `orderClauses`); the INT field can still be a **column** (works fine) and stays click-sortable in the result-table header. (This is why the contract search orders by STRING `ContractDate`, not the INT score.) Verified via `POST /core/rest/documents/search` with an `OrderClause`: STRING field → 200, INT field → 500 T00104.
  - **Nested / two-level `FieldAggregation` is NOT supported inside a search `hiddenRequest`** on this instance → same `T00104` on load. The `<property name="nested"><list><bean FieldAggregation…>` shape is documented (and works for a `HomeGraphPresenter` histogram) but breaks the search form. Use a **single-level** `FieldAggregation` in a search; put any two-level histogram on a `HomeGraphPresenter` home widget instead.
- **Criterion (filter input) beans** = `SimpleCriterionPresenter` (free: `model` Criterion = name/type/operator) or `FilterCriterionPresenter` (adds `displayOperatorSelector=false`, `forceMonovalued=true`, `mandatory` → renders as a single clean value field). Add a `description` `List<I18NLabel>` (EN/FR) to give the **filter input a human label/placeholder** (doc p…). The **filter label** comes from `description`; the **column header** comes from the **tag-class `displayName`** (§6) — both must be fixed for clean UX.
- **Indexing form** is mostly NOT free XML: tags **auto-render** (widget per tag, type from tag type), grouped via the class's **TagCategory** links, with readonly/mandatory on the class `tagReference` flags (all §6, class-model side). To scope a config bean to ONE class build a `com.flower.docs.domain.componentclass.IdBasedContext` (props: `id`=Id{value=classId}, `category`=DOCUMENT, `phase`=INSERT|MODIFY|READONLY). System-metadata relabel (e.g. "Name"→"NDA reference") = a `ComponentPropertiesConfiguration` of `FieldDefinition` beans bound to that context — but `componentProperties` and `componentActivityConfigurations` are **singletons in gui-solution.xml**; a second top-level bean of the same id **clobbers** all classes. Merge entries INTO the existing bean, never ship a duplicate.
- **Surfacing a search in the left-nav is NOT in the XML** — it's a **profile property** `search.template`. `/core/rest/teams` + `/core/rest/profiles` 404, BUT the profiles + their properties live in the **scope doc**, which IS editable via Core: ✅ **verified** the surfacing path end-to-end:
  - `GET /core/rest/scope/IRIS` → array → `[0].people.profiles[]`, each with `properties[]` of `{"name":..., "value":...}`.
  - **Update** = `POST /core/rest/scope/IRIS` (id in path, **array** body) → 200. (PDF only documents `get`, but POST/{id} works like tagclass; `POST /rest/scope` no-id → 500 "symbolic name must be unique" = create.) A **no-op write round-trips losslessly** — only `data.lastUpdateDate` changes — so a minimal additive edit is safe. **Back up the scope first**, and after writing, diff live-vs-backup to assert only your additions moved (auto-restore otherwise).
  - To surface a search: append `{"name":"search.template","value":"<beanId>()"}` to a profile's `properties` (note: value is **`beanId()` with EMPTY parens** — the label comes from the bean's `title` I18NLabel list, NOT from parens). Adding to all profiles → visible to every login. **Additive & reversible** (remove the entry + cache clear).
  - Same property pattern for the home dashboard (`home.widget.catalog`=`<SimpleWidgetCatalog beanId>`), the create `+` list (`sol-shortcuts` → `componentCreationShortcuts`), virtual-folder tabs (`tab.virtualfolder`), and `place.shortcut`.
  - After the scope write, **`DELETE /gui/rest/caches` + reload** for it to take. Verified live: the surfaced search shows in the "Searches" nav with its `title`, human column headers (from tag displayNames), clean criterion labels (from `description`), classid-scoped results, newest-first.
- **Deploy recipe:** upload XML → `POST /rest/files/tmp` (multipart) → create `GUIConfiguration` doc via `POST /rest/documents` with `data.classId:"GUIConfiguration"` + a `RegistrationOrder` tag (controls bean-id collision precedence; reserve a band per use case) → **`DELETE /gui/rest/caches`** (cookie session; purges `IRIS-GUIConfiguration`/`IRIS-Script`) → **full page reload**. Rollback = `DELETE /core/rest/documents/{id}` + cache clear + reload. (`files[].size:0` in the create response is a display quirk; content is present.)

## 10. Server-side event handlers = JS that runs in the Core JVM (Graal) ✅ — verified on IRIS

FlowerDocs reacts to operations (CREATE/UPDATE/DELETE/SEARCH/ADD_CONTENT/ANSWER/…) via **operation managers**. The scriptable one runs **JavaScript in the Core JVM via the embedded Graal engine** — server-side, no server access needed to deploy (it's just a document). This is the way to do **enrich-at-ingestion** (compute verdicts once on upload, store as tags → demos read cached tags, instant + cheap).

- **Subscription = an `OperationHandlerRegistration`-class document** (deploy via Core API). Config **tags**: `OperationHandler` (=`com.flower.docs.core.tsp.operation.script.ScriptOperationHandler` for JS; `…DroolsOperationHandler` for an Excel decision table; `…fulltext.FullTextOperationHandler` for indexing; or an **http(s) URL** for an external REST hook), `ExecutionPhase` (`BEFORE`|`AFTER`), `Action` (`CREATE`/`UPDATE`/`DELETE`/`SEARCH`/`ADD_CONTENT`/`ANSWER`/`ASSIGN`/`PROMOTE`/`REVERT`/`READ`), `ObjectType` (`DOCUMENT`/`TASK`/`FOLDER`/…), `Enabled` (bool), `Asynchronous` (bool — use **true** for slow AI work so it doesn't block the create), `StopOnException` (bool — **false** unless a BEFORE-gate must reject; only a BEFORE script can throw to abort), `RegistrationOrder`, plus Hook-only `InjectToken`/`Authorization`. The class' tagRefs: Enabled/OperationHandler/ExecutionPhase/Action/ObjectType/Asynchronous/RegistrationOrder/StopOnException/ReserveComponentsAsynchronous/InjectToken/Authorization.
- **Files on the doc:** (1) the handler **`.js`** as content; (2) optionally a filter file **named exactly `request`** = a serialized `SearchRequest` XML restricting which components trigger it (e.g. `classid EQUALS_TO NdaContract`) — **use this to stay shared-safe** so your handler only fires on your class, not every create on the shared instance.
- **Injected script vars:** `component` (the doc/task), `context` (`OperationContext` — `getUser()`, `getScope()`, phase), `request`/`response` (search ops), `logger` (SLF4J → Core log, not readable without server access — so make effects observable as **tags**). `util` = service access (`getDocumentService()`/`getUserService()`/`getFolderService()`, `create()`, `update(component)`, `changeClass()`, `createFact()`, `getAttachments(task)`, `search(request)`); `RuleUtil` (static) = in-memory `getTagValue`/`setTagValue`/`getClassId`/`setClassId`/`getStatus`/`setStatus`. BEFORE-phase tag writes persist with the operation; AFTER-phase needs `util.update(component)` to persist. Pre-imported builders: `ComponentBuilder`/`TagBuilder`/`CriterionBuilder`/`FilterClauseBuilder`/`SearchRequestBuilder`/`FactBuilder`/`ReferenceBuilder`/`ExceptionBuilder` + `Operators`/`Types`/`Lists`. Use new-handler conventions: `==` not `.equals()`, `String.valueOf(new Date().getTime())` not `Date.now()`.

## 14. Handler HTTP DELETE bug, taskclass recreate CONFIRMED, JSAPI navigation ✅ (2026-06-10, Ct module)
- **CRITICAL helper bug pattern — bodyless DELETE silently sent as GET**: the proven `http()` helper used in Graal handlers did `if (body != null) b.method(method, …) else b.GET()` — so EVERY bodyless non-GET call (task DELETE!) was actually a GET. Symptom: "DELETE returns 200 but the resource survives" (the 200 was the GET). Cost three full debugging iterations (blamed answer-commit races and token types first). Fix: `else if (method == 'GET') b.GET(); else b.method(method, HttpRequest.BodyPublishers.noBody())`. Audit any copy of this helper.
- **Taskclass-recreate hypothesis CONFIRMED** (was §13 OPEN ISSUE): after switching to a NEW once-created taskclass (`CtDeviationApproval2`) and repointing the registration filter + handler guard + task creation, ANSWER dispatch fires reliably (~5s, fresh reg ids + cache clear + ~45s wait as usual). Discipline: a deploy script must NEVER delete/recreate a taskclass; create it once via an explicit one-shot mode, verify-only on deploy; if the class must change, mint a NEW id.
- **§13 correction**: "deleting the answered task from within its own ANSWER handler needs retry/backoff" — unproven either way; those deletes were GETs (bug above). Working pattern (verified e2e): ANSWER handler only updates the other document (contract status); the document-UPDATE handler — a separate pipeline — deletes the answered task by id from a tag + clears the tag (idempotent, observable, no race with the answer commit).
- **Task search rows DO carry the component id**: `POST /rest/tasks/search` with `selectClause+max` returns `results[i].id` at the TOP LEVEL of each row (alongside `fields`) — no need to embed ids in names (softens §12's "the doc id is NOT a field"; check `.id` on document rows too). Note: an answered (undispatched or dispatched) task still shows `status: NEW` in search rows — status does NOT distinguish answered tasks; don't build worklist filters on it.
- **Client JSAPI navigation**: `JSAPI.get().navigateTo` does NOT exist (never documented — a hallucinated API; fails silently into error paths). Real API (docs p686/736): `JSAPI.get().getNavigationAPI().goToComponentPlace(category, id, confirmation)` with category `DOCUMENT|FOLDER|TASK|VIRTUAL_FOLDER`, plus `goTo(flower.docs.PlaceBuilder.build(placeToken), confirmation)` for place tokens (e.g. `storeTask(...)`, `search(...)`, `browse(id=…,leaf=…)`), `reload()`, `goBack(confirmation)`, `getWherePlace()`.

## 19. DOCX tracked-change editing inside a Core JS handler — verified end-to-end ✅ (2026-06-16, CtSuggestClause experiment)
Goal: a handler that injects a Word "suggestion mode" tracked change (`w:del`+`w:ins`) into a contract DOCX. Works — but every one of these bit us first:
- **NEVER read the content of the very doc the handler is firing on.** `GET /core/rest/documents/{id}/files/{fileId}/content` from inside a handler that triggered on that same doc's UPDATE → **500 `T00702` "Document files cannot be retrieved"** (the content blob is locked/being re-resolved by the triggering operation — cancel-and-replace re-POSTs `files[]`). Metadata GET on the same doc still works (reads the index), so `getFileId` succeeds and only the content GET fails — misleading. **Fix: read from a SEPARATE source doc that is not being updated, and WRITE the result to a NEW doc** (don't self-replace). The same external token retrieves that content fine from Node — it is NOT a token problem (minted-JWT vs `/authentication` token made no difference; both act as the same `system` user). Reading a different, idle doc from the handler works.
- **GraalVM handler scope: declare every `Java.type(...)` at MODULE scope, not inside the IIFE.** Helper functions defined outside the IIFE can't see IIFE-local `var`s → they throw `ReferenceError`, get swallowed by a `try/catch`, and you get a misleading downstream null (we chased a phantom `no-file-id`). Same top-level pattern as CtIngest.
- **The JS global `String` is bound to `java.lang.String` in this engine** → `String(n)` throws *"execute on java.lang.String failed: Message not supported"* (calling a string as a function). Use `('' + n)` to stringify.
- **`new (Java.type('java.lang.String'))(jsStr).getBytes()` fails** — GraalVM unboxes java.lang.String to a JS string with no `.getBytes()`. To get UTF-8 bytes for a hand-rolled multipart, wrap the `ByteArrayOutputStream` in an `OutputStreamWriter(baos, StandardCharsets.UTF_8)`; `w.write(text); w.flush();` before/after raw `baos.write(bytes)`.
- **AFTER-phase handlers re-fire on EVERY `util.update(component)`** (each tag write triggers another AFTER/UPDATE). The idempotency guard MUST treat ALL terminal states as stop conditions — `APPLIED`/`PROCESSING` **and** `ERROR:*`/`FATAL:*`. We initially let `ERROR` through → the handler re-ran on its own error write → **infinite loop that bumped the doc version ~1700× in a minute**. (Redeploy via `uxc push` rotates `_vN`/deletes the old registration → kills a runaway loop.) BEFORE-phase is NOT a fix here: in BEFORE the component holds the PRE-update state, so a freshly-set trigger tag reads as null and the guard exits before doing anything.
- **Hand-rolled multipart to `/core/rest/files/tmp` from the JVM works** (boundary `--B\r\nContent-Disposition: form-data; name="file"; filename="x.docx"\r\nContent-Type: …\r\n\r\n` + bytes + `\r\n--B--\r\n`, header `Content-Type: multipart/form-data; boundary=B`). Verified byte-for-byte against the Node FormData path: both store identical bytes (read back via `GET /rest/files/tmp/{id}`). `files:[{id:tmpId}]` then binds the content on **create** of a new doc (`POST /core/rest/documents` array body). `files[].size:0` in the echo is the usual display quirk — content is really there.
- **OOXML mechanics that actually render in Word:** parse `word/document.xml` namespace-aware; set revision attrs with `setAttributeNS(W_NS,'w:id'|'w:author'|'w:date',…)` (plain `setAttribute('w:id',…)` silently no-ops in a NS-aware DOM); `w:del` wraps the original runs with each `w:t`→`w:delText` (keep `xml:space="preserve"` via `setAttributeNS(XML_NS,…)`); `w:ins` holds a new `w:r/w:t`. The JAXP `Transformer` (INDENT=no) **preserves the `w:` prefix** (no `ns0:` rewrite) — verified. Re-zip with `[Content_Types].xml` first. Match a clause by concatenating a paragraph's `w:t` text and `indexOf(oldClause)` — but it only matches text that lives in ONE paragraph's runs (whole-paragraph clause replacement). (NB: the V4 NDA corpus DOCX carry junk `[trash]/*.dat` ZIP entries from their generator — harmless, Word ignores unreferenced entries; preserve them as-is.)
- **PRODUCTION SHAPE = "command document" pattern** (cleanest way to edit a contract's DOCX IN PLACE): a throwaway `CtSuggestionCommand` doc carries `CtTargetContractId` + the edit JSON; firing on ITS create lets the handler read+replace the TARGET contract (idle → no T00702) and then DELETE itself (`HttpRequest…DELETE()`). Verified: command self-deletes, target fileId changes, file gains the tracked change. A handler editing a *different* doc than the one it fired on is the general escape hatch from the self-reference lock.
- **Keep the original file across an in-place replace → enable per-class versioning + promote first.** The document class field is **`versioningMode`** (`NONE`|`MANUAL`|`AUTO`; default/absent = NONE). - **Word COMMENTS (annotations) via OOXML** (verified): a comment touches FOUR ZIP parts — append `w:comment` (with `w:annotationRef` + text run) to **word/comments.xml** (create the part if absent); insert empty sibling `w:commentRangeStart`/`w:commentRangeEnd` + a reference run (`w:commentReference`, AFTER rangeEnd) around the target paragraph's runs in **word/document.xml**; add a `Relationship` (type `.../comments`, Target `comments.xml`) to **word/_rels/document.xml.rels**; add an `Override` (PartName `/word/comments.xml`) to **[Content_Types].xml**. The `w:id` must match across comments.xml ↔ the range markers ↔ the reference. ⚠️ **CRITICAL namespace point the naive briefing got wrong:** `.rels` and `[Content_Types].xml` elements live in the OPC **default** namespaces (`…/package/2006/relationships`, `…/package/2006/content-types`) — with a namespace-aware DOM you MUST `createElementNS(<thatNS>, 'Relationship'|'Override')`, NOT `createElement(...)` (plain createElement emits a namespace-cancelling `xmlns=""` → Word rejects the part). Inject minimal `CommentText`/`CommentReference` styles into word/styles.xml if absent (corpus DOCX lack them; Word still shows the comment but the styles make the balloon render cleanly). New comments.xml is a NEW entries-map key → writeDocx writes it. Same `('' + id)` (not `String(id)`) GraalVM rule applies.
- **Batch review-package render (many findings, ONE pass)** + two comment-anchoring gotchas (both verified):

## §22 — In-handler text extraction: DOCX (zip) + native PDF via PDFBox (verified live 2026-06-26, IRIS)
- **The slow path:** `flowerDocsService.extractTextualContent(docId)` (the gateway extractor) is too slow for real
  PDFs — it blows the ~120 s gateway/render timeout on multi-page contracts. Extract text IN-JVM in the handler instead.
- **PDFBox IS on the OperationHandler classpath on IRIS** (confirmed by ingesting a native PDF end-to-end). Extract a
  PDF's text layer in-JVM, version-agnostically:
  ```js
  var PDFTextStripper = Java.type('org.apache.pdfbox.text.PDFTextStripper');
  var doc;
  try { doc = Java.type('org.apache.pdfbox.Loader').loadPDF(bytes); }                    // PDFBox 3.x
  catch (e) { doc = Java.type('org.apache.pdfbox.pdmodel.PDDocument').load(bytes); }      // PDFBox 2.x
  try { var s = new PDFTextStripper(); s.setSortByPosition(true); return '' + s.getText(doc); }
  finally { try { doc.close(); } catch (x) {} }
  ```
  `bytes` = the file content as a Java `byte[]` (from `HttpResponse.BodyHandlers.ofByteArray()` over `/rest/documents/{id}/files/{fileId}/content`). Wrap in try/catch returning `null` so the caller can fall back if PDFBox is ever absent.
- **DOCX in-JVM (no lib needed):** `new ZipInputStream(new ByteArrayInputStream(bytes))` → read `word/document.xml`
  → `replace(/<\/w:p>/g,'\n').replace(/<[^>]+>/g,'')` + unescape `&amp;/&lt;/&gt;/&quot;/&#39;/&#160;`. Fast (~secs).
- **Dispatch by file MAGIC BYTES, not extension** (uploads often lack/lie about extensions): read `bytes[0..3] & 0xff`
  — `25 50 44 46` = `%PDF` → PDFBox; `50 4B` = `PK` → DOCX/zip; else treat as plain text. (GraalVM lets you index a
  Java `byte[]` directly: `bytes[0] & 0xff`, and `bytes.length` works.) One `extractText(token,docId)` entry handles both.
- **Doc-level metadata under chunked ingestion:** when you chunk a large doc, no single AI call sees the whole thing,
  so doc-level fields (parties, counterparty, reference, term, covers) get dropped. Fix with ONE cheap extra prompt
  over just the doc HEAD (first ~2 chunks / ~5000 words — preamble + first articles is where these live). gpt-5.4-nano
  temp 0 returns them in ~2-3 s. Cheap, no verbatim reproduction, no refusal wall.

## §28 — Folder UPDATE from a handler: in-JVM does NOT persist → REST minted as `system` (SOLVED, verified 2026-07-04, fd.demo)
A ScriptOperationHandler can **create** a folder in-JVM (`util.getFolderService().create([folder])` persists — §27/p580 pattern), but **updating an existing folder from a handler does NOT stick**: neither `util.getFolderService().update(folder)` nor the generic `util.update(folder)` persisted a tag change. Verified TWICE — the ingest handler advancing an existing order's status on a follow-up email, and the ANSWER handler advancing the order on `DONE`: both no-op'd (no error, status unchanged). The component the handler fires ON updates fine via `util.update(component)`; a DIFFERENT folder does not. → To mutate an order/case **folder** from a handler (advance status on answer, etc.), use **Core REST with a JWT minted as the `system` identity**. **Mint `system`, not the answering user** — so the write is authorized regardless of who answered: a non-admin operator's own identity may lack rights on the order folder (ACL), and a handler triggered by arbitrary users must not depend on the caller's rights. This is correct-by-design and matches CT. **The gate is the token's PROFILES claim — CONFIRMED by A/B (2026-07-04):** mint two tokens identical except profiles, SAME `setId` (the answering user) in both, and write the folder — profiles `[SYSTEM_ADMIN,ADMIN]` (copied from the `system` user) → PERSISTS; profiles `[]` → silently does NOT persist (no 4xx observed, just no effect). So identity-as-`setId` is irrelevant; the write is authorised purely on the PROFILES claim. ⚠️ **The mint carries ONLY what you pass to `setProfiles()`** — group-derived authority is dropped. On fd.demo, `aescaffre` has `profiles:undefined, groups:["Admin"]` (admin via the **Admin group**), while `system` has `profiles:["SYSTEM_ADMIN","ADMIN"]`. So `setProfiles(aescaffre.getProfiles())` yields an EMPTY-profile, unauthorised token — even though `aescaffre`'s real `/authentication` login updates folders fine. ⚠️ **Why the discrepancy (decoded the JWTs to confirm):** `/authentication` RESOLVES group membership into effective profiles — `aescaffre`'s token payload is `profiles:["ADMIN","ALL_USERS"], groups:["Admin"]` — but `util.getUserService().get(login).getProfiles()` returns only the **declared** profiles (empty for a group-admin like `aescaffre`). So minting from `getProfiles()` drops the group-resolved authority; minting from the `system` user (which has DECLARED `[SYSTEM_ADMIN,ADMIN]`) is why it works. (Also verified: JWTs are HS256 signed with the raw-UTF8 secret bytes — a base64-decoded key is rejected 403; you can mint valid FlowerDocs tokens from Node this way.) → Always mint from a **profile-based** admin; the **`system`** user is the reliable source. (To faithfully mint AS a group-admin user you'd also have to copy groups onto the `AuthenticatedUser`, if it supports it — untested. Sync-handler call budget is tight — ~2 REST calls; a 4-call probe silently exceeded it, so observe results externally rather than tracing in-handler.) Note in-JVM services ALREADY run elevated (they call Core services directly), yet an in-JVM update of a component OTHER than the one the operation fired on still doesn't persist — that's a transaction-scoping limit, NOT a rights one; the REST bounce works because it commits as its own transaction. There is **no in-JVM `runAs`/impersonate primitive** in the scripting API (whole manual swept) — identity travels on the token, so you "become" another principal by minting its token (this IS the sudo). Exact recipe (verified): read the signing secret in-JVM from a config doc (§21); `var u = util.getUserService().get(['system'], true).get(0)` (system exists on `default` with `["SYSTEM_ADMIN","ADMIN"]`); build `AuthenticatedUser` with `setScope(context.getScope())`, `setId(context.getUser())`, **`setProfiles(u.getProfiles())`**; `new JWTTokenHelper().setSecretKey(secret).generate(au).getValue()` → `token:` header → GET `/rest/folders/{id}` → JSON.parse `[0]`, replace the tag → **POST `/rest/folders/{id}` with an ARRAY body** (`[folder]`). ⚠️ endpoint: the single-id path `POST /rest/folders/{id}` with `[folder]` works; the **collection `POST /rest/folders` 500s** for an update. **VERIFIED WORKING end-to-end 2026-07-04** on fd.demo `default`: answering DONE/NEED_INFO/CANCEL flips the order folder to VALIDATED/ON_HOLD/CANCELLED (this is exactly the CT ANSWER-handler pattern — CT also mints as `system`). This also proves AFTER/ANSWER/TASK handlers **do fire** on `default` (firing was never the problem; the mint identity was). NOT the in-JVM service. (Folder update via REST with a plain `/authentication` token works from a script too — it's specifically the in-JVM handler path that drops folder edits.) Related: async handlers don't execute at all on the bare `default` scope (no async worker, like Uxopian-AI) → register operation handlers **synchronous** there.

## §30 — Virtual folder as a FIRST-CLASS metadata object (case/order model) — full REST mechanics (verified 2026-07-07, fd.demo default)
A VF is not just a saved search — it's a metadata-carrying component you can use as the "case/order/dossier" object, aggregating its children by a **pivot tag** (no physical filing).
- **VF CLASS carries `tagReferences`** (a typed metadata schema) just like a folderclass — verified the server accepts `POST /rest/virtualfolderclass` with `tagReferences:[{tagName,multivalued,order,mandatory,readonly,technical}]`. The class also holds `searches:[{id,category:DOCUMENT|TASK,request:{filterClauses,orderClauses,aggregation,max}}]`. In a class search, a criterion value of **`${tags.<TagName>}`** is substituted per-instance from that instance's tag → one class serves every instance (e.g. `PoOrderRef EQUALS_TO ${tags.PoOrderRef}` aggregates the order's emails). ⚠️ VF-class search clauses use the **`type`** discriminator (`"type":"com.flower.docs.domain.search.AndClause"`), NOT `@class` (that's the /rest/documents/search dialect; uxc's fd.vfclass validator flags it).
- **VF INSTANCE = the object**, carrying tag VALUES. CRUD (learnings §15 + verified): create `POST /rest/virtualFolder` ARRAY `[{id,category:"VIRTUAL_FOLDER",name,data:{ACL,classId},tags:[{name,readOnly,value:[...]}]}]`; get `GET /rest/virtualFolder/{id}` → **array-of-1**; update `POST /rest/virtualFolder/{id}` ARRAY = **cancel-and-replace** (must re-send the whole component incl. ALL tags — GET, mutate, POST); delete `DELETE /rest/virtualFolder/{id}`. From a handler, same §28 rule: write via **REST minted as `system`** (in-JVM update of another component doesn't persist).
- **VF instances ARE searchable as a category** → the "all cases" overview + KPIs survive. `POST /rest/virtualFolder/search` with the **`@class`** search dialect (`{selectClause,filterClauses:[{"@class":"…AndClause",criteria:[{name:"classid",…,values:["PoOrder"]}]}],aggregation:{field,nested},max}`). ⚠️ **~1–2 s index lag** after create before a new instance is found (an instant search returns found:0; GET-by-id is immediate). Search by classid AND by any tag both work.
- **Why choose VF over a real folder for the case object:** children aggregate by the shared pivot tag (drop the whole `addChildren` + collision-tolerance mess), and the view is nicer. **Styling the VF view = a GUIConfiguration content-override** (§15: search bean id `content<Classid>VirtualFolder`, presentation-only) — NOT `registerForComponentChange` (that hook is for DOCUMENT/TASK forms; whether it fires on a VF instance view is unconfirmed — assume not, use the GUIConfig override + the VF's native tag rendering).
- **Task create over REST = `POST /rest/tasks` with NO trailing slash** (FD2026: `/rest/tasks/` → 404 "No endpoint"). ARRAY body `[{category:"TASK",name,assignee:"system",data:{classId,ACL},tags:[…],attachments:[{componentIds:[docId],id:"<childrenSlot>",category:"DOCUMENT",order:0}]}]`; no `workflow`/`id` needed (server auto-assigns id, `assignee:"system"` → participants). Task update = `POST /rest/tasks/{id}` ARRAY (cancel-replace). **Taskclass in-place update is binding-safe** (`POST /rest/taskclass/{id}` full-replace, e.g. to add tagReferences) — only DELETE+recreate breaks answer dispatch (uxc fd.taskclass = createOnly + inPlaceUpdate:true, so `uxc push` does the safe in-place update).
- **Design pattern — case/ticket processing status ≠ linear pipeline** (validated vs Zendesk/Freshdesk/O2C): one **processing status** (company-must-act vs waiting-external vs done) + a **who-we-await** field + a **multi-value activities** tag (the exception taxonomy) + an appended **state-log JSON** for stats. AI decides transitions from the new message + full history; **low-confidence/high-stakes → a human review task, never a silent auto-move** ("feel easier, not worse"). Aging on the waiting state ("depuis N j → à relancer") is the follow-up signal.

## §31 — ARender viewer URL + VF order-view UI (all Puppeteer-verified on fd.demo, 2026-07-07)
- **ARender viewer URL that RENDERS a doc**: `/{origin}/gui/plugins/arender/ARender.html?props=arender&scope=<scope>&currentId=<docId>`. The **`currentId`** param is the one that works (matches the gui-solution `arenderConfig` `documentIdParameter:currentId`). ✗ `uuid=b64_<btoa(docId)>` alone renders NOTHING; ✗ `documentId=<id>` errors to a sample empty.pdf. - **The doc MUST have a file.** ARender renders a *file*; a metadata-only doc (created via REST with tags but no `files:[…]`) → ARender shows **"An error occurred / Could not open document"** (looks like a 500 after a timeout). Paste-only emails (text in a tag, no file) can't be ARender'd — only uploaded PDFs/files. Seed demo emails WITH a file (`POST /core/rest/files/tmp` → `files:[{id:tmp}]`).
- **Native ARender pane on TASK views works out of the box** — open a task whose attachment (children slot) is a real doc-with-file and the platform auto-renders it in the right pane, with a multi-doc navigator across all attachments. The earlier "ARender doesn't open on tasks" was purely fileless attachments. No gui-solution edit was needed for this on fd.demo.
- **VF (folder/order) view is a two-pane layout with a native ARender viewer you can drive**: DOM `.row.content-row` = `.left-container.col-lg-4` (the metadata form + the card stack) · `.viewer` (a **~940px right pane containing an ARender `<iframe>`**, initially empty) · `.right-bar`. To render a doc in the right pane from a Script, set **`document.querySelector('.viewer iframe').src = <ARender currentId URL>`** — feels native, no overlay. (An injected fixed-position panel also works as a fallback where there's no `.viewer`.)
- **`registerForComponentChange` DOES fire on a VF instance (order) view** — the decorated cards render there. They land in the left-column card stack AFTER the native **"Metadata"** form (the auto-rendered editable tag form, foldable). Verified: cards reorder above Metadata.

Give the VF ONE content search of the primary child type (e.g. emails) — a second search (tasks) can make the content pane show the wrong category.
- **Client-side VF search is NOT available to the GUI session**: `POST /gui/rest/virtualFolder/search` → **404** (VF search lives only on `/core`, which needs a token, not the session cookie). Contrast: `/gui/rest/documents/{id}/files/content?index=0` DOES serve the file over the session (302→login if unauthenticated) — usable as a raw-PDF `<iframe>` fallback, though ARender is the product-correct viewer.

## §33 — Client scripts: `registerForComponentChange` accumulates across navigations — boot-once guard (verified 2026-07-07, fd.demo)

FD's GUI is a **SPA**. When the user navigates to a new document (without a full page reload), FD **re-executes all registered client scripts** (Script documents) for that document's class. Each execution calls the script's boot function, which calls `JSAPI.get().registerForComponentChange(fn)` again — **registrations are cumulative**: after N document navigations, there are N listeners registered. Same applies to `document.addEventListener` calls inside boot.

**Symptoms:** cards, action buttons, and overlays are duplicated (N copies rendered per component change event); dashboard overlays stay stuck (N concurrent searches create N overlapping overlays; only one resolves); "send to specialist" creates N tasks per click.

**Fix — boot-once guard via a window-level flag:**
```javascript
function boot() {
  if (window.__ctwBooted) return;   // already registered — skip re-registration
  window.__ctwBooted = true;
  // ... registerForComponentChange, document.addEventListener, etc.
}
```
`window.__ctwBooted` persists for the page's lifetime (cleared on full reload, which clears all accumulated listeners too). Only the FIRST execution of the script registers. Subsequent navigations cause the script to re-execute but `boot()` returns immediately.

**Companion fix — shared live-card registry:** If the script's internal card-dedup map is a local variable (not window-level), the SECOND execution's listener has its own empty map and always re-decorates (defeating the `document.contains(prevCard)` guard). Fix:
```javascript
var __ctwCard = window.__ctwCard || (window.__ctwCard = {});
```
With the boot-once guard, there is only one listener, so this companion fix is only needed as a safety net for pathological re-execution scenarios (e.g., hot-reloads, unusual GWT state management).

## §26 — Duplicated GUI links: surfacing value equivalence + multi-profile union (verified 2026-07-08, fd.demo)
- **Symptom**: every package link (search templates, VF tabs, home-widget entries like the AI chat) appeared MULTIPLE times in the FD 2026 GUI.

- `uxc doctor --dups` now scans every profile for same-name properties whose values share the same leading identifier token (`X()` vs `X` vs `X(args)`) — the duplicate-link detector.

## §27 — Handler RegistrationOrder bound + firing-probe protocol (verified 2026-07-08, fd.demo FD 2026)

## §34 — Installing a Pulse Marketplace addon on a scope + the Uxopian AI (Xopia) assistant wiring (verified fd.demo, 2026-07-08)
- **Marketplace auth**: the `uxc mp` API key (`uxmk_…`, 45 chars) lives in the **macOS keychain** service **`uxc/marketplace`**, account `aescaffre` — NOT in `~/.uxopian/marketplace.json` (that file holds only `url` + `maintainer`). Pass it per-call via the **`UXC_MARKETPLACE_TOKEN`** env override so it's never written to disk: `UXC_MARKETPLACE_TOKEN="$(security find-generic-password -s uxc/marketplace -a aescaffre -w)" uxc mp ls`. Browse needs auth (anon → 401).
- **Install onto a scope**: `uxc mp install <slug>[@ver] --target <t> [--force]` = download + sha256-verify + pre-flight + ordered push. Pre-flight classifies each object: `adopted` (byte-matches server → skipped) / `collision` (same id, DIFFERENT content). Default **aborts on any collision**. **`--force` = the incoming package wins → updates the colliding objects to the package's values** (and skips the already-matching ones) — exactly "install and make existing values match the package." ⚠️ `import`/`mp install` extracts a **working copy into `cwd/<code>/`** (e.g. `./uxoai/`) — run it from a **neutral dir** (`~/projects`) so it doesn't drop a foreign package inside your current one. Receipts land on the target: `uxc installed --target <t>`.
- **The Uxopian AI packages**: `uxoai-flowerdocs` (generic) and **`uxoai-flowerdocs-fddemo`** (fd.demo-ready) = **12 self-configuring UI scripts** (chat/robot button, admin shortcut, translate, quick-prompts — endpoints derived at runtime from page origin + live scope) **+ a `Gateway` Route doc** (classId `Route`, tags `Path=/gateway/**`, `URL=http://gateway-service.iris.svc.cluster.local:8085` — the verified fd.demo in-cluster gateway, valid for every fd.demo scope; the fd.demo variant pre-fills this one per-instance value). On fd.demo/`default` the Gateway route already existed with the correct URL, so only the 8 stale scripts needed updating (4 already matched).
- **"Handlers work but the Xopia assistant doesn't respond" → almost always STALE/absent front-end scripts (or a missing Gateway route / LLM provider), NOT the AI backend.** Handlers run server-side and mint their own system JWT to Core (§28) — independent of the browser chat. The chat needs: (a) the 12 UI scripts current, (b) the Gateway route, (c) an LLM provider+key. Fix for a silent assistant = (re)install the package with `--force` (refresh the scripts) + hard-reload.
- **Verify the AI**: `uxc doctor --ready --target <t>` gates L0.platform / L0.versions / L0.ai (gateway + prompt count) / L0.llm (providers). `uxc doctor --ai-smoke` does a live gateway→provider→model round-trip. ⚠️ **`--ai-smoke` can report FAIL when the model merely *refuses* its canned probe** ("I'm sorry, but I can't comply…") even though the pipeline is healthy (a real completion came back) — treat a refusal as "pipeline OK, model cautious", confirm with a real prompt/model choice. ⚠️ **`uxc run <promptId>` (v0.7.1) streams via gateway SSE and may hang without printing captured output** for some prompts — `doctor --ai-smoke` is the reliable backend check.
- **Scope policy clarified (user, 2026-07-08): "never IRIS" = never the OLD server `iris.demos.uxopian.com`** (targets `iris`/`fd`/`ctm`/`ctnaked`/`gerflor`). The **IRIS *scope* on `fd.demo`** is fine to modify **when explicitly asked** — it's just a scope on the current server. Don't over-restrict fd.demo scopes to read-only; only the legacy server is off-limits.
- **Upgrading a package version with removals (uxc ≥0.10.0 "upgrade pruning", verified 2026-07-10 upgrading uxoai 1.0.0→1.3.0 which dropped the `refreshToken` script)**: `mp install <slug> --target <t> --force` over an installed receipt detects objects present in the old version but absent in the new and offers to **prune** them. ⚠️ **Non-interactive requires the named flag `--yes-removals`** — a `yes |` stdin pipe does NOT satisfy it (it's a flag, not a Y/n prompt). ⚠️ **Gotcha: without `--yes-removals` the removals are SKIPPED but the RECEIPT still advances to the new version** → the orphan is stranded, and re-running install now says "reinstalling (already on the target)" and no longer re-detects the removal. **So pass `--yes-removals` on the upgrade install itself.** To prune a **stranded orphan** afterward: `uxc rm <kind>/<id> --server` fails with "unknown resource" because the new version's registry no longer lists it → first **`uxc adopt <kind> <id> --target <t>`** (pull it into a package registry), then **`uxc rm <kind>/<id> --server --target <t>`** (deletes server-side + tombstones locally + clears caches).
- **Package dependencies (uxc ≥0.9.0 check-and-guide)**: a package can declare deps (e.g. 1.3.0 declares `llm: requires *`) checked against the target's **install receipts** (`uxc installed`) — NOT the live server state. A scope with functional LLM providers configured manually (no `uxopian-ai-default-providers-set` receipt) trips a **false-negative** "NOT INSTALLED". Verify the real requirement (`uxc ls ai.llm --target <t>` → providers present) then override with **`--ignore-dependencies`** (safe — it only skips the check, doesn't touch anything). ⚠️ Do NOT "satisfy" it by installing the providers-set package when providers already work — that can reset the provider configs / drop the admin-panel API keys and break the assistant.

## §36 — The handler-rotation window can DOUBLE-fire, not just lose events (verified fd.demo/IRIS, 2026-07-11)

§12 records that events during the ~45 s registration-propagation window are LOST for the NEW
registration. The mirror image is also real: shortly after a rotation (`_vN` deploy + delete of
`_vN-1`), a single CREATE was observed dispatching **two full executions** of the ingest pipeline
(two `START_MS` generations ⇒ duplicate clause sets are prevented by deterministic ids, but
duplicate REVIEW/APPROVAL tasks and duplicate render commands/annotated copies are not).
Uploads ~70 s and ~25 min after a rotation double-fired; a run ~40 min after was exactly-once.
Operational rules: (a) leave real traffic — and `uxc test` e2e runs — a settling period after a
handler push; (b) when duplicate tasks/annotations appear, check whether a rotation just
happened before suspecting the handler logic; (c) handlers should make their derived-object ids
deterministic per SOURCE (not per execution) wherever duplicates would hurt.
